## 二、相关技术背景

### 2.1 Linux字符设备驱动框架

Linux将所有设备抽象为文件，字符设备通过 `/dev/xxx` 节点以字节流方式访问。驱动程序通过注册 `file_operations` 结构体向内核声明其支持的操作：

```
用户进程
   │  open("/dev/kbuf") / read() / write()
   ▼
VFS（虚拟文件系统）
   │  根据主设备号找到对应 file_operations
   ▼
kbuf_driver（本课设驱动）
   │  操作内核缓冲区，通过等待队列同步
   ▼
内核内存（kmalloc / 静态数组）
```

### 2.2 生产者/消费者模型

生产者/消费者模型是经典的并发同步问题：

- **生产者**：生成数据，放入共享缓冲区；缓冲区满时阻塞等待；
- **消费者**：取出数据，处理并消费；缓冲区空时阻塞等待；
- **共享缓冲区**：本课设采用环形队列（Circular Buffer）管理多个固定大小的槽位。

环形队列示意图：

```
  slot[0]  slot[1]  slot[2]  slot[3]  slot[4]  slot[5]  slot[6]  slot[7]
  ┌──────┬──────┬──────┬──────┬──────┬──────┬──────┬──────┐
  │ FULL │ FULL │ FULL │ FREE │ FREE │ FREE │ FREE │ FREE │
  └──────┴──────┴──────┴──────┴──────┴──────┴──────┴──────┘
              ▲                    ▲
          read_pos=2           write_pos=3
          (消费者从这里读)      (生产者向这里写)
                   count = 3
```

### 2.3 等待队列（Wait Queue）

Linux内核等待队列是实现进程阻塞和唤醒的核心机制：

| 接口 | 作用 |
|------|------|
| `init_waitqueue_head(&wq)` | 初始化等待队列头 |
| `wait_event_interruptible(wq, cond)` | 若cond为假则睡眠，可被信号中断 |
| `wake_up_interruptible(&wq)` | 唤醒队列中所有等待的进程 |

使用可中断版本（interruptible）是驱动开发的最佳实践，允许用户通过 Ctrl+C 终止阻塞的进程。

### 2.4 /proc 文件系统

`/proc` 是Linux内核向用户空间暴露信息的伪文件系统。本课设通过 `seq_file` 接口在 `/proc/kbuf_status` 中实时显示缓冲区状态，无需任何额外的ioctl命令即可监控驱动内部状态。

## 四、核心数据结构

### 4.1 缓冲槽 kbuf_slot

```c
struct kbuf_slot {
    char   data[BUFFER_SIZE];   /* 数据存储区，4096字节 */
    size_t len;                  /* 本槽实际有效数据长度 */
};
```

每个槽位独立存储一条消息，长度可变（最大4096字节）。

### 4.2 设备对象 kbuf_dev

```c
static struct kbuf_dev {
    struct kbuf_slot      slots[NUM_BUFFERS]; /* 环形队列槽位数组 */
    int                   read_pos;           /* 下一个消费槽位索引 */
    int                   write_pos;          /* 下一个生产槽位索引 */
    int                   count;              /* 当前已满槽位数量 */
    wait_queue_head_t     read_wq;            /* 消费者等待队列 */
    wait_queue_head_t     write_wq;           /* 生产者等待队列 */
    struct mutex          lock;               /* 保护以上所有共享状态 */
    struct cdev           cdev;               /* 内核字符设备对象 */
    dev_t                 devno;              /* 设备号（主+次） */
    struct class         *cls;               /* 设备类（用于自动创建节点） */
    struct device        *dev;               /* 设备实例 */
    struct proc_dir_entry *proc_entry;        /* /proc条目指针 */
} kbuf;
```

**字段说明：**

- `slots[]`：固定大小的槽位数组，构成环形队列的存储后端；
- `read_pos / write_pos`：生产/消费位置指针，通过取模运算实现环绕；
- `count`：当前满槽数量，是阻塞判断的核心条件变量；
- `read_wq / write_wq`：两个独立等待队列，分别服务于读者和写者，避免不必要的跨角色唤醒；
- `lock`：mutex互斥锁，确保对队列状态的原子操作；
- `cdev / devno / cls / dev`：Linux字符设备注册所需的标准字段。

## 五、关键机制与实现

### 5.1 阻塞写（生产者）

```c
static ssize_t kbuf_write(struct file *filp, const char __user *ubuf,
                           size_t count, loff_t *ppos)
{
    if (mutex_lock_interruptible(&kbuf.lock))
        return -ERESTARTSYS;

    while (kbuf.count == NUM_BUFFERS) {          /* 缓冲区满 */
        mutex_unlock(&kbuf.lock);
        if (filp->f_flags & O_NONBLOCK)
            return -EAGAIN;                       /* 非阻塞：立即返回 */
        if (wait_event_interruptible(kbuf.write_wq,
                kbuf.count < NUM_BUFFERS))
            return -ERESTARTSYS;                  /* 被信号中断 */
        if (mutex_lock_interruptible(&kbuf.lock))
            return -ERESTARTSYS;
    }

    copy_from_user(kbuf.slots[kbuf.write_pos].data, ubuf, count);
    kbuf.slots[kbuf.write_pos].len = count;
    kbuf.write_pos = (kbuf.write_pos + 1) % NUM_BUFFERS;
    kbuf.count++;

    mutex_unlock(&kbuf.lock);
    wake_up_interruptible(&kbuf.read_wq);         /* 唤醒等待的读者 */
    return count;
}
```

**流程说明：**
1. 加锁后检查缓冲区是否已满；
2. 若满且为非阻塞模式，返回 `-EAGAIN`；
3. 若满且为阻塞模式，释放锁后进入 `write_wq` 睡眠，等待消费者消费后唤醒；
4. 醒来后重新加锁，再次检查（while而非if，防止虚假唤醒）；
5. 将用户数据拷贝进槽位，更新指针和计数，解锁，唤醒读者。

### 5.2 阻塞读（消费者）

```c
static ssize_t kbuf_read(struct file *filp, char __user *ubuf,
                          size_t count, loff_t *ppos)
{
    if (mutex_lock_interruptible(&kbuf.lock))
        return -ERESTARTSYS;

    while (kbuf.count == 0) {                     /* 缓冲区空 */
        mutex_unlock(&kbuf.lock);
        if (filp->f_flags & O_NONBLOCK)
            return -EAGAIN;                        /* 非阻塞：立即返回 */
        if (wait_event_interruptible(kbuf.read_wq,
                kbuf.count > 0))
            return -ERESTARTSYS;
        if (mutex_lock_interruptible(&kbuf.lock))
            return -ERESTARTSYS;
    }

    size_t len = min(count, kbuf.slots[kbuf.read_pos].len);
    copy_to_user(ubuf, kbuf.slots[kbuf.read_pos].data, len);
    kbuf.read_pos = (kbuf.read_pos + 1) % NUM_BUFFERS;
    kbuf.count--;

    mutex_unlock(&kbuf.lock);
    wake_up_interruptible(&kbuf.write_wq);         /* 唤醒等待的写者 */
    return len;
}
```

读流程与写流程完全对称，当缓冲区为空时消费者阻塞于 `read_wq`，由生产者写入数据后唤醒。

### 5.3 阻塞与非阻塞对比

| 场景 | 阻塞模式（默认） | 非阻塞模式（O_NONBLOCK） |
|------|------------------|--------------------------|
| 读，缓冲区空 | 进程睡眠，等待生产者写入 | 立即返回 -EAGAIN（errno=EAGAIN） |
| 写，缓冲区满 | 进程睡眠，等待消费者读取 | 立即返回 -EAGAIN（errno=EAGAIN） |
| 正常读/写 | 完成操作，返回字节数 | 完成操作，返回字节数 |

### 5.4 互斥锁与等待队列的配合

等待队列的使用必须遵循"释放锁→睡眠→重新加锁"的固定模式，否则会产生竞态条件：

```
线程A（生产者）               线程B（消费者，缓冲区空）
    lock()
    检查 count==0 → 真
    unlock()                 ← 必须先释放锁
                             lock()
                             写入数据，count++
                             unlock()
                             wake_up(read_wq)
    wait_event(...)          ← 此时已满足条件，不会永久睡眠
    重新 lock()
    消费数据
```

`wait_event_interruptible` 内部在睡眠前会再次检查条件，因此即使唤醒发生在睡眠之前，条件已满足，进程也不会错过唤醒信号。

### 5.5 /proc 状态接口

采用 `seq_file` 接口实现，线程安全（持锁读取状态）：

```c
static int kbuf_proc_show(struct seq_file *m, void *v)
{
    mutex_lock(&kbuf.lock);
    seq_printf(m, "=== kbuf driver status ===\n");
    seq_printf(m, "Total slots    : %d\n", NUM_BUFFERS);
    seq_printf(m, "Slot size      : %d bytes\n", BUFFER_SIZE);
    seq_printf(m, "Full slots     : %d\n", kbuf.count);
    seq_printf(m, "Free slots     : %d\n", NUM_BUFFERS - kbuf.count);
    seq_printf(m, "Read  position : %d\n", kbuf.read_pos);
    seq_printf(m, "Write position : %d\n", kbuf.write_pos);
    mutex_unlock(&kbuf.lock);
    return 0;
}
```

### 5.6 模块初始化流程

```
alloc_chrdev_region()     分配设备号（主设备号由内核动态分配）
        │
cdev_init() + cdev_add()  向内核注册字符设备及file_operations
        │
class_create()            创建设备类（/sys/class/kbuf/）
        │
device_create()           自动创建 /dev/kbuf 节点（无需手动mknod）
        │
mutex_init()              初始化互斥锁
init_waitqueue_head()     初始化读/写两个等待队列
        │
proc_create()             创建 /proc/kbuf_status 条目
        │
pr_info("kbuf: loaded")   打印加载日志（可通过dmesg查看）
```

卸载时按相反顺序清理所有资源（proc条目→设备节点→设备类→cdev→设备号）。

### 6.2 test_producer.c

生产者程序以阻塞模式打开 `/dev/kbuf`，循环写入带时间戳的消息，每次写入后延迟指定毫秒数。支持命令行参数控制消息数量和写入速度，便于演示生产速度快于消费速度时的阻塞行为。

### 6.3 test_consumer.c

消费者程序以阻塞模式打开 `/dev/kbuf`，循环读取消息并打印。消费延迟可设置为大于生产延迟，演示消费者超前消费导致的阻塞等待（队列为空时消费者睡眠）。

### 6.4 test_nonblock.c

分三阶段测试非阻塞模式：
1. 空队列时非阻塞读 → 立即返回 `EAGAIN`；
2. 逐槽写入直到队列满 → 第9次写时返回 `EAGAIN`；
3. 非阻塞读依次取出8个槽位 → 全部成功；最后一次读返回 `EAGAIN`。

## 九、总结与体会

### 9.1 设计总结

本课程设计实现了一个完整的Linux内核字符设备驱动，核心功能如下：

1. **多缓冲区环形队列**：8个固定大小（4096B）槽位，生产者/消费者分别通过 `write_pos` 和 `read_pos` 独立操作，互不干扰；
2. **等待队列同步**：`read_wq` 和 `write_wq` 两个独立队列，精确唤醒目标角色，避免惊群；
3. **双模式支持**：阻塞模式实现流量控制，非阻塞模式支持事件驱动架构；
4. **proc状态接口**：`seq_file` 安全地向用户空间暴露实时队列状态，便于调试；
5. **资源管理**：模块卸载时完整清理所有内核对象（cdev、class、device、proc条目），无内存泄漏。

### 9.2 学习体会

通过本次课程设计，深刻理解了以下内容：

- **"设备即文件"的本质**：用户空间的 `read()/write()` 系统调用最终映射到驱动的 `file_operations`，VFS层完成了这个透明的转换；
- **等待队列的正确使用**：必须在释放锁之后才能进入睡眠，醒来后重新加锁，并用 `while` 而非 `if` 防止虚假唤醒；
- **内核编程的严谨性**：内核代码中的一个空指针或竞态条件会直接导致系统崩溃，必须对每个错误路径进行处理并保证资源正确释放；
- **proc文件系统的实用性**：相比ioctl，`/proc` 接口更易于调试，`cat /proc/kbuf_status` 即可查看驱动内部状态，无需额外工具。
